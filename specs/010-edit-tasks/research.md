# Phase 0 Research: Edit Tasks

All decisions follow from the clarifications (content-only fields, scope
immutable, recurrence editable, pending-only). The headline finding is that the
chosen scope needs **no backend change**.

## Decision 1: No backend / migration change

**Decision**: Editing mutates only `name`, `task_date`, `description`, `urgent`,
`recurrence` and (when recurrence is first enabled) `series_id`. It never changes
`owner_id` or `nucleus_id`.

**Rationale**: The `tasks_update` RLS policy already authorizes a member to
update a task they can see, and the `immutable_ownership` trigger only forbids
changing `owner_id`/`nucleus_id` (which we don't). The outbox UPDATE payload in
`syncEngine` already sends exactly these columns. So an edit syncs through the
existing path with zero schema, RLS, trigger or type changes.

**Alternatives considered**: Allowing scope changes — rejected at the spec
clarification (would require relaxing the trigger + careful RLS). Out of scope.

## Decision 2: Model the edit as a pure `applyEdit` (test-first)

**Decision**: Add `domain/edit.ts`:
```ts
applyEdit(task: Task, parsed: ParsedNewTask, now: string): Task
```
It returns a new Task with `name`, `taskDate`, `description`, `urgent`,
`recurrence` taken from `parsed`; `updatedAt = now`; **preserves** `id`,
`ownerId`, `nucleusId`, `completedAt`, `completedBy`, `createdAt`; and sets
`seriesId`:
- recurrence enabled and `task.seriesId === null` → a new series id (passed in / generated by caller to keep the function pure — see Decision 4),
- recurrence enabled and series id present → keep it,
- recurrence null → leave `seriesId` as-is (irrelevant without recurrence).

**Rationale**: Principle IV names "state-transition logic" as test-first. A pure
function makes the preservation guarantees (identity/owner/scope/completion) and
the recurrence/series handling executable documentation, and keeps the
repository method thin.

**Alternatives considered**: Inlining the merge in the repository — harder to
test and easy to accidentally clobber a preserved field.

## Decision 3: Reuse `parseNewTask` for validation

**Decision**: The edit submit runs the same `parseNewTask` used by creation
(non-blank name; valid `YYYY-MM-DD` date; "dueDate" recurrence anchor requires a
date). The form passes the edited fields; `nucleusId` is irrelevant in edit and
ignored by `applyEdit`.

**Rationale**: Identical validation rules to creation (spec FR-004) with no
duplicate schema (Principle II). Already unit-tested.

**Alternatives considered**: A separate edit schema — needless duplication.

## Decision 4: Series id generation stays at the impure boundary

**Decision**: `crypto.randomUUID()` for a newly enabled series is generated in
the repository (`editTask`) and passed into `applyEdit`, keeping `applyEdit`
deterministic/pure. Same pattern as `createTask` (which also generates the id in
the repo).

**Rationale**: Keeps the domain function pure and testable; randomness lives in
the data layer.

## Decision 5: Generalize the form (`CreateTaskForm` → `TaskForm`)

**Decision**: One form component for both create and edit:
- `initial?: { name; taskDate; description; urgent; recurrence }` — pre-fills in
  edit mode (empty in create).
- `submitLabel` — "Añadir tarea" (create) / "Guardar" (edit).
- Scope `<select>` is rendered **only in create mode** (scope is immutable on
  edit, spec FR-003).
- `onSubmit(input)` + `onCancel`.

`TasksPage` uses it in create mode (behavior unchanged); the edit callers use it
pre-filled.

**Rationale**: Avoids a second near-identical form (Principle VII) and guarantees
the edit view matches the create view the user already knows (spec FR-002).

**Alternatives considered**: A distinct edit form — duplication and drift.

## Decision 6: Edit-in-place, in both list and deck

**Decision**:
- **List (`TaskItem`)**: a pending task shows an "Editar" control; activating it
  swaps the row content for the `TaskForm` (pre-filled). Save/cancel returns to
  the normal row.
- **Deck (`TaskCard`/`TaskDeck`)**: the top card shows an "Editar" action
  (alongside the existing skip/stop); activating it replaces the card area with
  the `TaskForm` for that task until save/cancel.

Only pending tasks expose the control (spec FR-007); `editTask` is also a no-op
on a completed task as a defensive guard.

**Rationale**: Edit-in-place is the most intuitive and needs no global "editing
task" state lifted across the tree; each surface manages its own local toggle.

**Alternatives considered**: Lifting an `editingTaskId` to `TasksPage` and
rendering one form at the top — more wiring through `TaskGroups`/`GroupSection`
for no UX gain.

## Decision 7: Reordering and propagation are automatic

**Decision**: After `editTask` writes to Dexie, the existing `liveQuery`
re-emits and `groupTasks`/ordering re-place the task (date cleared → "ya";
urgent toggled → reorders). For group tasks, the outbox push + Realtime deliver
the change to other members. No extra work.

**Rationale**: The edit is just another task write; the reactive read path and
sync already handle the consequences (spec SC-001/SC-002).
